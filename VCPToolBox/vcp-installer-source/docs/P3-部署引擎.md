## 一、P3在项目中的位置

P3负责**VCP前后端的实际部署流程**：git clone仓库、npm install依赖、pip install依赖、生成config.env配置文件、生成启动脚本、下载NewAPI。

**依赖关系**：P1（下载引擎）+ P2（Portable运行时）→ **P3（部署引擎）**

P3的输入是P2准备好的Portable运行时路径，输出是完整可运行的VCP环境。

## 四、git_ops.rs — Git克隆与拉取

### 公开接口（3个）

```rust
/// 克隆仓库（浅克隆 --depth 1）
/// 目标存在且是git仓库 → 自动git pull
/// 目标存在但不是git仓库 → 报错
pub fn git_clone(
    git_exe: &Path,      // Portable Git的git.exe路径
    repo_url: &str,      // 仓库URL
    dest: &Path,         // 目标目录
    env_path: &str,      // PATH环境变量
) -> Result<()>

/// 判断目录是否为git仓库（检查.git子目录存在）
pub fn is_git_repo(dir: &Path) -> bool

/// 拉取最新代码（--ff-only快进合并）
pub fn git_pull(
    git_exe: &Path,
    repo_dir: &Path,
    env_path: &str,
) -> Result<()>
```

### 内部辅助函数

```rust
/// 格式化Command执行结果的stdout+stderr为可读字符串
fn format_process_output(output: &std::process::Output) -> String
```

### 核心逻辑

1. **git_clone流程**：
   - `git_exe.exists()` 前置检查 → 快速失败
   - `dest.parent()` 不存在则 `create_dir_all` 自动创建父目录
   - 目标已存在 + is_git_repo → 调用 `git_pull`
   - 目标已存在 + 非git仓库 → `anyhow::bail!` 报错
   - 新建克隆：`git clone --depth 1 {repo_url} {dest}`

2. **git_pull流程**：
   - `repo_dir.exists() && is_git_repo` 双重校验
   - `git -C {repo_dir} pull --ff-only`
   - 非零退出码 → `format_process_output` 输出详细错误

3. **错误处理**：统一使用 `anyhow::Context` 的 `with_context` 提供链式错误上下文

## 五、npm_ops.rs — npm依赖安装与启动

### 公开接口（2个）

```rust
/// npm install（安装项目依赖）
pub fn npm_install(
    node_dir: &Path,      // Portable Node.js目录
    project_dir: &Path,   // 项目目录（含package.json）
    env_path: &str,       // PATH环境变量
    use_mirror: bool,     // 是否使用淘宝镜像
) -> Result<()>

/// npm start（启动项目，返回子进程句柄）
pub fn npm_start(
    node_dir: &Path,
    project_dir: &Path,
    env_path: &str,
) -> Result<std::process::Child>
```

### 内部辅助函数（4个）

```rust
/// 获取node可执行文件路径（Windows: node.exe）
fn node_executable(node_dir: &Path) -> PathBuf

/// 获取npm可执行文件路径（Windows: npm.cmd）
fn npm_executable(node_dir: &Path) -> PathBuf

/// 构建npm Command（Windows用 cmd /C npm.cmd 解决兼容性）
fn new_npm_command(node_dir: &Path) -> Command

/// 统一设置npm相关环境变量（PATH + npm_config_fund=false + npm_config_audit=false）
fn apply_npm_env(cmd: &mut Command, node_dir: &Path, env_path: &str)
```

### 核心逻辑

1. **npm_install流程**：
   - `node_exe.exists()` + `npm_exe.exists()` 双重前置检查
   - `new_npm_command` → Windows下用 `cmd /C npm.cmd`（直接调用npm.cmd有兼容性问题）
   - `apply_npm_env` 设置PATH + `npm_config_fund=false` + `npm_config_audit=false`（静默赞助和审计提示）
   - **镜像方案：`--registry=https://registry.npmmirror.com` 参数级注入**
     - ⚠️ 文档原方案是 `npm config set registry`（污染全局配置），代码改为参数级注入，不污染用户npm全局配置
   - 非零退出码 → `format_process_output` 详细报错

2. **npm_start流程**：
   - 同样的 `new_npm_command` + `apply_npm_env`
   - `cmd /C npm.cmd start`
   - 返回 `Child` 进程句柄（由上层管理生命周期）

```rust
/// 在文件内容中替换指定key的值（KEY=OLD_VALUE → KEY=NEW_VALUE）
/// 只替换第一次出现的匹配（防重复写入）
fn replace_env_value(content: &str, key: &str, value: &str) -> String

/// 过滤环境变量值中的\r\n字符（防配置注入）
```

   - `project_dir.exists()` 前置检查
   - 读取 `config.env` 模板文件
   - 已存在config.env → 备份为 `config.env.bak`
   - `replace_env_value` 替换每个配置项
   - `sanitize_env_value` 过滤 `\r` `\n` 字符（**安全性改进：防配置注入**）

3. **download_newapi流程**：
   - 调用P1的 `downloader::get_github_release_url` 获取最新Release
   - 安全过滤：排除文件名含"setup"（不区分大小写）或不以".exe"结尾的asset
   - 调用P1的 `downloader::download_file` 下载
   - 进度通过 `progress_tx` 回报

### 🌟 三泛型步骤执行器（代码最大亮点）

文档原设计中，每个步骤都有相似的 StepStarted → 执行 → match Ok/Err → StepCompleted/StepFailed 模式，导致大量重复代码。实际代码抽象为三个泛型辅助函数：

```rust
/// 阻塞任务执行器（git clone/npm install/pip install等CPU密集型操作）
/// 用tokio::task::spawn_blocking包装，含JoinError处理
async fn run_blocking_step<F>(
    tx: &Sender<ProgressEvent>,
    step_index: usize,
    description: &str,
    errors: &mut Vec<String>,
    f: F,
) -> bool
where F: FnOnce() -> Result<()> + Send + 'static

/// 同步轻量任务执行器（config.env生成等快速操作）
fn run_sync_step<F>(
    tx: &Sender<ProgressEvent>,
    step_index: usize,
    description: &str,
    errors: &mut Vec<String>,
    f: F,
) -> bool
where F: FnOnce() -> Result<()>

/// 异步任务执行器（download_newapi等需要await的操作）
async fn run_async_step<F, Fut>(
    tx: &Sender<ProgressEvent>,
    step_index: usize,
    description: &str,
    errors: &mut Vec<String>,
    f: F,
) -> bool
where F: FnOnce() -> Fut, Fut: std::future::Future<Output = Result<()>>
```

三个函数统一行为：
- 发送 `StepStarted` 事件
- 执行闭包/Future
- 成功 → `StepCompleted` + 返回true
- 失败 → `StepFailed` + errors.push + 返回false
- `run_blocking_step` 额外处理 `JoinError`（spawn_blocking panic不崩溃整体）

### 其他内部辅助函数

```rust
/// 发送跳过事件 + 返回false
fn skip_step(tx, step_index, reason, errors) -> bool

/// 发送日志事件
fn send_log(tx, message)

/// 发送StepStarted + 标记为pending
fn pending_step(tx, step_index, description)
```

### run_installation 执行流程

```
1. 准备阶段
   ├── 初始化 installed_components: Vec<String>
   ├── 初始化 errors: Vec<String>
   └── 构建 env_path（P2的build_env_path）

2. 后端部署（if config.components.backend）
   ├── run_blocking_step: git_clone VCPToolBox
   ├── 记录 error_count_before
   ├── run_blocking_step: npm_install（git成功才执行，否则skip_step）
   ├── run_blocking_step: pip_install（git成功才执行，否则skip_step）
   ├── run_sync_step: generate_config_env
   ├── 如果 error_count == error_count_before → push "VCPToolBox后端"
   └── run_sync_step: generate_start_backend_bat

3. 前端部署（if config.components.frontend）
   ├── run_blocking_step: git_clone VCPChat
   ├── run_blocking_step: npm_install（git成功才执行）
   ├── run_blocking_step: pip_install（git成功才执行）
   ├── 如果全部成功 → push "VCPChat前端"
   └── run_sync_step: generate_start_frontend_bat

4. NewAPI部署（if config.components.newapi）
   ├── run_async_step: download_newapi
   └── 成功 → push "NewAPI"

5. 结果汇总
   ├── success = errors.is_empty()
   ├── start_script = 检查bat文件实际存在才返回路径
   └── 返回 InstallResult
```

**关键设计：容错与降级**
- git失败 → npm/pip自动skip（不是报错，是StepSkipped事件）
- 单个组件失败不影响其他组件继续
- `error_count_before` 模式精确判断每个组件的整体成功状态
- `installed_components` 按实际成功追踪，而非照抄config

## 十、与文档原版的主要差异

| 项目 | 文档原版 | 实际代码 |
|------|----------|----------|
| npm镜像方案 | `npm config set registry`（污染全局） | `--registry=` 参数级注入（不污染） |
| npm调用方式 | 直接调用npm | `cmd /C npm.cmd`（Windows兼容性） |
| 步骤执行 | 每步重复StepStarted/Completed/Failed | 三泛型执行器 run_blocking/sync/async_step |
| 失败处理 | 简单push error | skip_step + StepSkipped事件 + error_count_before模式 |
| 配置安全 | 直接写入 | sanitize_env_value 过滤\r\n防注入 |
| 脚本健壮性 | 基础bat | setlocal + if not exist检查 + call npm |
| installed追踪 | 照抄config.components | 按实际成功追踪 |
| start_script | 直接返回路径 | 检查文件实际exists才返回 |
